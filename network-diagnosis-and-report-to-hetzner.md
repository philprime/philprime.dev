<p>If you are experiencing network problems with your server, then please send us a support request via <a href="https://console.hetzner.com/support" target="_blank" rel="nofollow noopener noreferrer">Hetzner Console</a>. Select "Technical" and set the subcategory to "Public Network issue" or "Private Network issue". Make sure to select the affected server in your support request.</p>
<h2 id="server-is-not-reachable" style="position:relative;"><a href="#server-is-not-reachable" aria-label="server is not reachable permalink" class="headerlink---gatsby before"><svg aria-hidden="true" height="20" version="1.1" viewBox="0 0 16 16" width="20"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Server is not reachable</h2>
<p>If your server is not reachable, you can try to restart it yourself. To do this, open Hetzner Console, select the respective project and navigate to the affcted server. In the server menu bar, go to "Power" and select "Power off". Once it is off, you can restart the server by selecting "Power On". If that doesn't work, it is possible that we blocked your server. For more information about such situation, please also see our  <a href="/robot/dedicated-server/troubleshooting/guideline-in-case-of-server-locking/">Guideline for server locking</a>.</p>
<p>If you still have the problem, please send us a ticket via <a href="https://console.hetzner.com/support" target="_blank" rel="nofollow noopener noreferrer">Hetzner Console</a>. Select "Technical" and set the subcategory to "Public Network issue" or "Private Network issue". Make sure to select the affected server in your support request. In the description, mention "Server is not reachable".</p>

<p>If you are experiencing packet loss, then we will need some evidence of this. Simple statements such as "my ping is bad" or "there is packet loss to my server" are unfortunately not enough for an error analysis.</p>
<p>Please do a trace (in both directions) with at least 200 packets with a tool like <code>MTR</code> or <code>WinMTR</code>. <p>Please follow these guidelines to create traces that are useful for our technicians:</p>
<ul>
<li>Send at least 200 packets.</li>
<li>Do each trace in both directions.  That means you need to do it from your local endpoint (PC, Laptop etc.) to the server and from the server back to your endpoint.</li>
<li>You can find the IP address of your local endpoint using an online tool, like <a href="https://ip.hetzner.com" target="_blank" rel="nofollow noopener noreferrer">https://ip.hetzner.com/</a>.</li>
</ul>
<p>You can use the following command to create an MTR with Linux or MacOS:</p>
<div class="gatsby-highlight" data-language="bash"><pre class="language-bash"><code class="language-bash"><span class="token function">mtr</span> <span class="token parameter variable">-s</span> <span class="token number">1000</span> <span class="token parameter variable">-r</span> <span class="token parameter variable">-c</span> <span class="token number">200</span> <span class="token operator">&lt;</span>TARGET-IP_OR_DOMAIN<span class="token operator">&gt;</span></code></pre></div>
<p>To do this, run the above command on both systems and replace the placeholder with the correct remote IP address or domain:</p>
<div class="gatsby-highlight" data-language="bash"><pre class="language-bash"><code class="language-bash"><span class="token punctuation">[</span>user@SERVER<span class="token punctuation">]</span>$ <span class="token function">mtr</span> <span class="token parameter variable">-s</span> <span class="token number">1000</span> <span class="token parameter variable">-r</span> <span class="token parameter variable">-c</span> <span class="token number">200</span> <span class="token operator">&lt;</span>CLIENT-IP_OR_DOMAIN<span class="token operator">&gt;</span>

<p>The test usually takes about 3-4 minutes.</p>
<p><strong>Important:</strong></p>
<ul>
<li>If your MTR shows the last hop to be unreachable, presumably the installed system or your local router is set to ignore ICMP requests due to security reasons. Nevertheless, the MTR can be used to investigate the connection.</li>
<li>The hops of an MTR show the process for the specific connection. So your MTR might look quite different than the following examples.</li>
</ul>
<p>These are problems that might become clear in test results: incorrect routing, very high latency on one of the network hops, or packet loss, which then leads to a re-transmission of those packets.
But please be careful; there can be three types of packet loss:</p>
